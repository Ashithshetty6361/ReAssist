# ReAssist Interview Preparation Guide

*A technical preparation sheet for discussing the ReAssist intelligence engine, covering architecture, operations, trade-offs, and scalability.*
## Q1: Walk me through what ReAssist does.
ReAssist is a research intelligence engine designed to automate the initial discovery and ideation workflow for researchers. The core of the system is a highly orchestrated 7-agent pipeline where specialized agents sequentially search academic papers, summarize them, synthesize literature, identify gaps, and propose novel methodologies. I also built a strict CoT (Chain-of-Thought) baseline agent to empirically compare against the multi-agent pipeline. Overarching this is the AgenticOps routing layer which algorithmically models incoming queries to dictate whether they deserve the expensive 7-agent treatment or the cheaper CoT single-prompt based on computational necessity.

## Q2: Why multi-agent instead of one big prompt?
Decomposing the task into a multi-agent system creates a manageable, debuggable architecture where each agent has a single strict responsibility. By utilizing a technique I call context slicing, I strictly limit the data passed to each agent using a `required_inputs` array—meaning an idea generation agent doesn't need to read the raw abstracts, only the synthesized gaps. Furthermore, separation enables resilient, granular error handling: if the search agent fails, I can retry just that node rather than blowing up a massive prompt. The obvious tradeoff, however, is that 7 distinct API calls fundamentally increases systemic latency and API expenditure compared to a single comprehensive prompt.

## Q3: How do you evaluate quality?
Evaluation is anchored fundamentally in human annotation regarding qualitative validity. I curated 15 difficult test queries and manually graded the outputs across three axes: Coverage, Specificity, and Actionability. Human evaluation guarantees that the novelty and feasibility of a research hypothesis are actually grounded in reality rather than just sounding plausible to a generative model. The pipeline loads these scores via a CSV when available, but if it encounters an un-annotated query, it seamlessly defaults to a deterministic structural check written in Python to ensure the output schema is at least comprehensively populated.

## Q4: What is AgenticOps and why does it matter?
AgenticOps is the operational discipline of actually running agentic systems in production without exploding budgets. The unsolved problem in AI engineering right now is that throwing multi-agent swarms at every user query is completely financially unsustainable. My solution was to build a routing layer that algorithmically scores incoming queries on complexity and precision (out of a 6-point index). If the score is low, it deflects the work to a cheap, capable baseline CoT model rather than triggering a multi-agent cascade. The system generates hard evidence of cost savings out of the box by dumping its calculations to `routing_log.jsonl`.

## Q5: How does your system reduce API costs?
The system utilizes three primary mechanisms to actively suppress API expenditure. First is context slicing; the `_slice_context()` method ensures that downstream agents literally cannot access massive upstream text blocks unless explicitly requested in `required_inputs`, immediately capping exponential token usage growth. Second, the AgenticOps router deflects roughly 60% of basic informational queries straight to the CoT baseline, dropping the execution cost from $0.015 down to $0.004 on `gpt-3.5-turbo`. Finally, the summarization layer utilizes an explicit token chunking mechanism capping documents strictly at 2000 tokens during its asynchronous scatter-gather phase.

## Q6: What model are you using and why?
The pipeline currently defaults to using `gpt-3.5-turbo` via the OpenAI SDK because it executes the JSON schema extractions exceptionally well while remaining extremely aggressively priced per token. The system allows an instantaneous backend toggle to `gpt-4` for intense production deployments. Because choosing to use `gpt-4` pushes the multi-agent pipeline cost from roughly $0.015 up to $0.45 per run, `gpt-4` is only justified if the research domain requires extreme logical extrapolation where the cheaper model's reasoning capabilities historically break down.

## Q7: What would break in production?
If this hit true production scale tomorrow, four specific things would instantly break. First, there is absolutely zero rate limiting, meaning a user spamming the search button would instantly blow through Semantic Scholar or OpenAI API quotas; I would fix this by putting Redis rate-limiting decorators over my routes. Second, there is no response caching; duplicate queries recalculate the exact same data. I'd add a semantic cache leveraging Redis to bypass LLM calls entirely. Third, there is no user authentication, which translates to a blank check for API billing; JWT auth integrating with something like Clerk would solve this. Fourth, it currently operates completely locally without a worker queue; scaling to multiple users means I would immediately need to migrate the agent pipeline into Celery background tasks so the HTTP requests don't time out.

## Q8: How is this different from AutoResearch or SciAgent?
While projects like AutoResearch focus massively on pure paper discovery chains, and SciAgent leans heavily into massive LLM context windows, ReAssist fundamentally differentiates itself via its explicit measurement and operational routing frameworks. I treated the AI agents not just as magic boxes, but as computational nodes that need to be aggressively monitored, benchmarked against a CoT baseline, and optionally bypassed via the AgenticOps router if the query overhead isn't justified.

## Q9: If you had 2 more weeks what would you add first?
I would immediately implement a Retrieval-Augmented Generation (RAG) extension leveraging ChromaDB. Right now, the multi-agent pipeline relies heavily on the summarizer to inject context, meaning it inevitably suffers from lossy compression over long research papers. By generating vector embeddings for the actual full-text PDFs and storing them locally in ChromaDB, the specific technical nuances of a paper could be pulled directly into the Synthesizer and Gap-Finder agents via similarity search, drastically increasing the specificity of the final ideas.

## Q10: What did you learn building this?
Actually engineering this from scratch taught me four vital lessons. First, multi-agent frameworks are incredibly fragile if they do not enforce rock-solid JSON schema outputs; I spent more time handling parsing failures than prompt tuning. Second, throwing more AI at a problem generally just makes it slower, not smarter; the CoT baseline empirically proved that complex orchestrations are frequently unwarranted. Third, `asyncio` combined with `ThreadPoolExecutor` is an absolute mandatory requirement for building modern AI tools, since I dropped latency significantly by parallelizing the summarization agent with 5 concurrent workers. Finally, implementing custom deterministic retries over LLM endpoints (`_run_agent_with_retry` with its 2-second backoff) is a requirement, not a nice-to-have, because temporary 502s from OpenAI will immediately nuke non-resilient pipelines.

## Hardest Interview Question
**Question:** If your router decides a query is 'simple', why run an LLM prompt at all instead of just relying on the direct Semantic Scholar API return? Aren't you overengineering a baseline?

**Answer:** That is a completely fair criticism. The reality is that the CoT baseline shouldn't simply be 'cheaper AI'—in many cases, the optimal route for a zero-complexity query is pure deterministic software. If I were designing this for strict enterprise scale, the router shouldn't just toggle between highly-expensive AI and moderately-expensive AI; it should have a third fallback tier that skips the LLM layer entirely and just formats the JSON payload directly from the raw database or API result, eliminating AI costs entirely when summarization isn't explicitly requested.
